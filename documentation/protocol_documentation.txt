meshTalk — Protocol Documentation
=================================

State: current runtime
Date: 2026-04-28

RU
--
1. Протокольный стек
- Весь meshTalk-трафик работает на текущем v3-стеке.
- Для meshTalk используется единый актуальный wire-формат.
- Стек состоит из двух уровней:
  1. внешний зашифрованный контейнер `envelope_v3`;
  2. внутренние relay/control кадры `relay_protocol`.

2. Внешний контейнер `envelope_v3`
Назначение:
- единый шифрованный транспортный контейнер для data и control payload;
- привязка сообщения к `msg_id` через AAD;
- единый формат для всего v3-трафика.

Формат:
- `version` = 3 (1 байт)
- `type` = 1 (1 байт)
- `msg_id` (8 байт)
- `nonce` (12 байт)
- `ciphertext + tag` (AES-256-GCM)

AAD:
- `[version, type] + msg_id`

3. Внутренний слой `relay_protocol`
Header содержит:
- `magic`, `version`, `frame_type`, `ttl`, `epoch_slot`
- `msg_id`, `relay_token`, `return_token`
- `frag_index`, `frag_total`, `body`

Назначение:
- `msg_id`: идентификатор логического сообщения;
- `ttl`: ограничение пересылки;
- `relay_token`: токен назначения;
- `return_token`: токен обратного подтверждения;
- `frag_index` / `frag_total`: разбиение на части.

4. Типы кадров
- `DATA`
- `HOP_ACK`
- `END_ACK`
- `TOKEN_ADV`
- `TOKEN_WITHDRAW`
- `CAPS_REQ`
- `CAPS`
- `REKEY1`, `REKEY2`, `REKEY3`

5. Доставка
- Исходящее сообщение разбивается на части.
- Каждая часть отправляется отдельным `DATA` кадром.
- На каждый `DATA` сосед возвращает `HOP_ACK`.
- После полной сборки конечный узел возвращает `END_ACK`.

6. Relay-модель
- Любой meshTalk-узел relay-capable.
- Пересылка управляется локальным `RelayState`.
- Решение о форварде принимается локально, без глобальной карты сети.

7. Отдельный путь `Primary`
- `group:Primary` использует обычный Meshtastic `TEXT_MESSAGE_APP`.
- Это отдельный путь, не relay-v3.

EN
--
1. Protocol stack
- All meshTalk traffic runs on the current v3 stack.
- meshTalk uses a single current wire format.
- The stack has two layers:
  1. outer encrypted `envelope_v3`;
  2. inner relay/control frames from `relay_protocol`.

2. Outer container `envelope_v3`
Purpose:
- one encrypted transport container for both data and control payloads;
- binds messages to `msg_id` through AAD;
- one format for all v3 traffic.

Format:
- `version` = 3 (1 byte)
- `type` = 1 (1 byte)
- `msg_id` (8 bytes)
- `nonce` (12 bytes)
- `ciphertext + tag` (AES-256-GCM)

AAD:
- `[version, type] + msg_id`

3. Inner layer `relay_protocol`
Header contains:
- `magic`, `version`, `frame_type`, `ttl`, `epoch_slot`
- `msg_id`, `relay_token`, `return_token`
- `frag_index`, `frag_total`, `body`

Purpose:
- `msg_id`: logical message identifier;
- `ttl`: forwarding limit;
- `relay_token`: destination token;
- `return_token`: return-path token;
- `frag_index` / `frag_total`: fragment numbering.

4. Frame types
- `DATA`
- `HOP_ACK`
- `END_ACK`
- `TOKEN_ADV`
- `TOKEN_WITHDRAW`
- `CAPS_REQ`
- `CAPS`
- `REKEY1`, `REKEY2`, `REKEY3`

5. Delivery
- Outgoing messages are split into fragments.
- Each fragment is sent as a `DATA` frame.
- Each `DATA` frame gets a `HOP_ACK` from the next hop.
- Once fully assembled, the destination returns `END_ACK`.

6. Relay model
- Every meshTalk node is relay-capable.
- Forwarding is controlled by local `RelayState`.
- Next-hop decisions are local, without a global route map.

7. Separate `Primary` path
- `group:Primary` uses plain Meshtastic `TEXT_MESSAGE_APP`.
- It is separate from relay-v3.
